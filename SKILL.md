# 日本語技術ブログ執筆スキル

日本語の技術ブログ記事を、プラットフォームの慣習・日本語特有の文体・SEO・読者体験を踏まえて
高品質に執筆するための指針です。

---

## ターゲットプラットフォーム

記事を書く前に、どのプラットフォームかを確認してください。分からない場合は汎用スタイルで書きます。

| プラットフォーム | 文体の傾向 | 特記事項 |
|---|---|---|
| **Zenn** | です・ます調が主流 / 技術的に深い | Markdown + emoji OK。`:::message` など独自記法あり |
| **Qiita** | です・ます調 / 実用重視 | タグ5つまで。`:::note` 記法あり |
| **note** | です・ます調 or 柔らかい地の文 | コードより読み物寄り |
| **個人ブログ** | ユーザーの指定に従う | |
| **社内ドキュメント** | だ・である調も可 | 読者が同僚なので専門用語の説明は短く |

---

## 記事の基本構造

以下のテンプレートを土台にしてください。必要に応じてセクションを追加・削除します。

```
# [タイトル] — 動詞または価値提案を含む 30〜50 字が理想

## はじめに（TL;DR を含む）
- 「この記事では〜します」という一文で目的を明示
- 対象読者・前提知識を 2〜3 行で示す
- 記事の要点を箇条書きで 3〜5 点（スキャン読みに配慮）

## 背景 / なぜこれが必要か（省略可）
- 課題や動機を簡潔に

## [本文セクション 1] — 具体的なタスク名で命名
## [本文セクション 2]
## [本文セクション N]

## まとめ
- 記事で達成したことを 3〜5 行で振り返る
- 次のステップや関連記事へのリンクを添える

## 参考文献 / 参考リンク
```

## 文体ガイドライン

### ですます調（推奨デフォルト）
- 語尾：〜です・〜ます・〜でしょう
- 疑問文：〜でしょうか？ / 〜ませんか？
- 柔らかさと明確さのバランスが取りやすい

### だ・である調（技術論文・仕様ドキュメント寄りの記事）
- 語尾：〜だ・〜である・〜といえる
- 引用ブロックや注釈内は混在しても構わない

### 共通ルール
- 一文は 40〜60 字を目安に。長い場合は分割する
- 接続詞「しかし」「また」「なお」は文頭に置いて段落の流れを明示する
- 英数字・アルファベットは半角、カタカナは全角を徹底する
  - ✅ `Docker` / `API` / `JSON`
  - ❌ `Ｄｏｃｋｅｒ` / `ＡＰＩ`
- 英語の技術用語はカタカナ変換より英語原文を優先する（`container` → コンテナ より `container` のまま文中に使うほうが検索しやすい）
- 体言止めは箇条書き内のみ。本文中では多用しない

## タイトル設計

良いタイトルは「何をする記事か」を一目で伝えます。

**パターン例：**

| パターン | 例 |
|---|---|
| 方法型 | `〇〇を△△する方法` |
| 入門型 | `〇〇 入門：△△から始める` |
| 比較型 | `〇〇 vs △△：□□で選ぶポイント` |
| 解決型 | `〇〇 で △△ が起きたときの対処法` |
| まとめ型 | `〇〇 を使いこなす 5 つのコツ` |

**SEO のコツ：**
- 検索されやすいキーワード（技術名・エラーメッセージ等）をタイトル前半に入れる
- 「〜してみた」は検索性が低い。具体的な動詞に置き換える
- 数字があると CTR が上がりやすい（`5 つの方法` / `3 分でわかる`）

## コードスニペットのガイドライン

````
```言語名
# 必ずシンタックスハイライトのために言語名を指定する
コード本体
```
````

- コードブロックの直前に 1 行の説明文を置く（「以下のコマンドで〜します。」など）
- 長いコードには行コメントで日本語の補足を入れる
- 出力例は別のコードブロックとして分ける（言語名は `bash` や `text`）
- ファイル名・パスはインラインコード `` `path/to/file` `` で示す

## 読者体験の設計

### 導線
- 記事冒頭に「対象読者」を明示する（例：「React の基礎がわかる方向け」）
- 長い記事は目次（`## 目次`）を入れるか、各セクションに「このセクションでわかること」を一行添える

### 強調・注記
| 目的 | Zenn 記法 | 汎用 Markdown |
|---|---|---|
| 情報・メモ | `:::message` | `> **📝 メモ：**` |
| 注意 | `:::message alert` | `> **⚠️ 注意：**` |
| ポイント強調 | `**太字**` | `**太字**` |

### 画像・図
- `alt` テキストは日本語で内容を説明する（`![Dockerコンテナのアーキテクチャ図](...)` など）
- キャプションは画像の直下に斜体か小さい文字で付ける

---

## よくある落とし穴と対処

| 落とし穴 | 対処 |
|---|---|
| 前置きが長すぎて本題に入らない | はじめに→背景→本題の順を守り、背景は 3〜5 行に収める |
| コードだけ貼って説明がない | コードの前後に「何をするコードか」「実行結果はどうなるか」を書く |
| 読者の前提知識を過小・過大評価 | 「はじめに」で対象読者を宣言し、初出の専門用語はリンクか一言説明を入れる |
| まとめが「以上です」だけ | 記事の価値を再確認し、次のアクション（関連記事・公式ドキュメント）を示す |
| 英語混じりで文体が揺れる | カタカナか英語原文かを記事内で統一する |

## 出力フォーマット

ユーザーから記事の執筆を依頼されたときは：

1. **確認フェーズ**（情報が不足している場合のみ）
   - トピック・テーマ
   - 対象プラットフォーム
   - 想定読者レベル（初級 / 中級 / 上級）
   - 記事の長さの目安（短め 1000〜1500 字 / 標準 2000〜3000 字 / 詳細 3000 字以上）

2. **執筆フェーズ**
   - 上記テンプレートと文体ガイドラインに従って完全な記事を生成する
   - プラットフォーム固有の記法（Zenn の `:::message` 等）を適切に使用する
   - コードが含まれる場合は必ず言語名付きのコードブロックを使う

3. **後処理の提案**（記事生成後に一言添える）
   - タイトルの改善案（2〜3 パターン）
   - 追加できるセクションや改善点の簡単なメモ
